Mais pourquoi ?

De un, allez tenter de comprendre la démarche de quelqu'un qui fait ce genre de commits, 6 mois après, en vous disant "mais oui, ce changement là, c'était dans ce commit !"

De deux, la plupart des projets Open source demandent une certaine rigueur, que ce soit niveau testing ou conventions de commits.

Aussi, peut-être aimez-vous faire vos revue de code en procédant "commit par commit", surtout pour les gros changements. Avec du code éparpillé dans tous les commits, c'est impossible de procéder comme ça. Vous avez peut-être déjà rencontré une situation où vous commentez une modification dans un commit qui disparaît dans le suivant ?

Enfin, à partir des messages de commits, on peut faire d'autres choses... comme générer des changelogs. Du coup on ouvre nos messages de commits à un public plus important que juste les développeurs, et tous doivent comprendre sans lire le code, si tant est qu'ils y aient accès.

Je ne vais pas m'étaler là-dessus, mais avec quelques conventions pour les noms de commits, il est possible de générer de beaux changelog comme celui que vous voyez.

A partir de là, on se doit d'écrire nos commits correctement...

Du coup, je reviens à mon équipe. La manière dont on fonctionne est la suivante.

On a une branche principale, qui dans notre cas est master, mais qui peut différer, par exemple si vous faites du Git Flow où ce sera develop... Mais l'idée principale est là, on a une branche partagée, sur laquelle personne ne commit directement.

  • On tire une branche
  • On fait un ou plusieurs commits
  • On soumet sa branche à une revue de code

Ensuite depuis la branche partagée, on tire une branche personnelle. C'est notre branche à nous, on fait ce qu'on veut dessus... On peut faire des bêtises, c'est pas grave, c'est notre branche.

Sur cette branche, on va développer nos fonctionalités, on va faire des commits... Bien formés tant qu'à faire.

Et une fois satisfaits, on soumet notre code à une revue de code pour demander à l'équipe de merger cette branche, dans la branche principale. Cela peut être une Pull Request sur GitHub, une Merge Request sur Gitlab, etc.

Comme je veux uniquement modifier le contenu du commit, j'utilise l'option --no-edit, cela permet de modifier uniquement le contenu sans modifier le message de commit. Il ne va juste pas ouvrir l'éditeur de texte qui permet de le modifier.

Qu'est-ce qu'on a fait ?

On est partis d'une branche contenant des commits A et B.

On a réécrit le dernier commit, c'est important de le souligner. Il n'a donc plus le même sha1. On passe de B à B'.

Sur Git, les commits, entre autres, sont identifiés de manière unique par leur sha1 (vous savez, la longue chaîne de caractères générée).

Aux yeux de Git, les commits B et B' sont donc totalement différents.

  • # Bien entendu il faudra « force push »
git push -f origin ma-branche

Les branches locale et distante ont divergé, et la plus à jour est la branche locale. Comme on est sur une branche personnelle, on peut sans danger écraser les modifications distantes.

  • Deux étapes

L'opération se fait maintenant en deux étapes...

Donc là c'est un peu plus compliqué mais vous allez voir, rien d'impossible.

A noter qu'il existe plusieurs méthodes pour arriver à nos fins. J'en présente une, celle qui me paraît la mieux adaptée à la plupart des situations.

  • 1. Créer un commit

  • de FIXUP

Première étape, on va créer un commit qui contiendra nos modifications. Mais pas n'importe quel type de commit...

Un commit de fixup !

  • git commit --fixup A

Je spécifie le commit à "fixup", en l'occurence ici A.

C'est un commit qui fait office de "patch" du commit à modifier. Il se rajoute en haut de la pile des commits. On l'appelle ici Af.

Ici, A, B et Af représentent les sha1 des commits en question, c'est à dire leur identifiant unique. On leur attribue un nom court pour plus de compréhension...

  • 2. Réécrire l'historique

  • REBASE

Deuxième étape: je dois intégrer mon commit de fixup dans le commit à modifier. Pour ça, il faut réécrire l'historique.

On va alors utiliser le Rebase.

Donc on veut partir de notre première étape (le fixup), et aller intégrer le commit de fixup dans notre commit A.

A sera donc réécrit en A'.

Enfin, B sera réécrit en B'. Git change son sha1 parce-que son parent aura changé. D'ailleurs, il est possible d'obtenir des conflits en réécrivant l'historique, par exemple si une modification ajoutée à Af est en conflit avec le contenu du commit B.

  • git rebase -i A~ --autosquash
  • -i : mode interactif
  • A~ : le parent de A
  • --autosquash : placer le commit de fixup au bon endroit

Oula, mais c'est quoi cette commande barbare ?

C'est simple, on va la découper :

-i: Le mode intéractif nous permet de choisir comment réécrire les commits: les renommer, les réordonner, changer leur contenu... plein de choses sont possibles.

A~: Quand on rebase, on se replace dans le passé pour faire comme si on n'avait pas encore créé les commits que l'on va réécrire. Du coup, si on veut réécrire l'historique à partir (et en incluant) A, il faut se replacer, non pas sur A mais juste avant. C'est ce que fait le tilde: il désigne le parent de A. En général, ajouter un tilde à un sha1 désigne le parent du commit désigné par le sha1.

--autosquash: Cela permet de replacer le commit de fixup directement au bon endroit pendant la procédure interactive.

Démo !

  • # Ou un alias git dans mon .gitconfig
[alias]
  fixup = !sh -c 'SHA=$(git rev-parse $1) \
       && git commit --fixup $SHA \
       && git rebase -i --autosquash $SHA~' -
# Utilisation : git fixup {sha1}

Pour faciliter la procédure, on peut écrire un alias Git.

Si on a oublié les conseils précédents, à savoir « force push si tu as modifié ta branche personnelle locale », on risque de récupérer une ancienne version de notre code, qui tentera de fusionner avec notre code local, légèrement modifié. Dans certains cas, il y aura même des conflits.

Voyons ce qu'il s'est passé

- En local, on avait réécrit des commits. Notre branche contient les commits A' et B'.

- Sur le repository distant, on a encore les commits dont on voulait justement se débarrasser. Pourquoi vouloir les récupérer en faisant un pull ?

On rappelle que les nouveaux commits sont totalement différents des anciens aux yeux de Git. Du coup que tente de faire Git... Il tente de les garder !

Git va donc essayer de satisfaire tout le monde et va créer un commit de merge pour garder tous les commits. Il est même possible d'obtenir des conflits...

Et si vous être plus malins et que vous avez utilisé la stratégie rebase (pull -r), voilà le résultat sur la droite. C'est même encore pire parce-que Git va essayer de replacer A' et B' au dessus de A et B. donc il va les réécrire encore une fois en A'' et B''.

Dans les deux cas, c'est loin d'être ce que l'on veut obtenir. On va retrouver dans l'historique plusieurs fois les mêmes messages de commits...

  • # Il fallait « force push »…
git push -f origin ma-branche

Encore une fois, on travaille sur notre branche personnelle, donc on a le droit.

Je ne détaille pas ici les solutions pour se sortir de cette situation, c'est au cas par cas... L'idée est d'annuler le commit de merge ou le rebase, ou alors de supprimer les commits en trop.

Du coup, je vais devoir supprimer un commit. En l'occurrence ici c'est le dernier.

Cela paraît plutôt simple (on va utiliser la commande reset), mais voyons les différentes options que nous offre Git. Pour comprendre ces options il faut comprendre comment Git gère nos sources.

Imaginons nous sur notre repository Git, en local, ayant notre fichier file.txt en version 1.

On distingue 3 notions différentes pour la gestion des sources.

- Le Working Directory: c'est ce qui se trouve effectivement sur votre disque dur : ce que vous voyez et pouvez modifier dans votre éditeur favori.

- L'index: Lorsque Git indexe les fichiers (git add), il crée dans le dossier .git un ensemble d'objets contenant le contenu des fichiers. Il crée aussi d'autres objets pour sauvegarder la hiérarchie et le nom de chaque fichier indexé. Tout cela forme un snapshot, prêt à être commité. D'ailleurs, un commit est un objet qui référence un snapshot.

- HEAD: C'est notre position actuelle sur Git. C'est à dire le commit sur lequel nous pointons. En d'autres termes, c'est un pointeur vers le commit en haut de la pile lorsque l'on lance un git log. Et donc par exemple si je me place sur master, alors HEAD, master et le dernier commit de master référencent exactement la même chose.

Imaginons que je modifie file.txt. Il passe en version 2.

Git détecte une différence entre le Working directory et l'index. Il me le notifie. C'est la fameuse zone rouge de mon git status.

A ce stade, la modification n'est pas indexée par Git, donc si je décide de faire un commit, ce dernier ne la contiendra pas.

Maintenant via un git add, je décide d'ajouter ma modification à Git. Elle devient indexée.

Git est heureux, l'index et le Working directory sont en phase. Par contre, HEAD et l'index ne le sont pas. Git me le notifie. C'est la fameuse zone verte de mon git status.

Cette fois, si je décide de faire un commit, la modification sera contenue dans celui-ci.

Inversement, si je lance un git reset sur mon fichier, Git supprime la modification de son index. On est revenu à l'étape précédente.

Mais revenons à l'étape "j'ai indexé ma modification"...

Je lance un git commit. Git est à nouveau heureux, le Working directory, l'index et HEAD sont au même point. Ils contiennent tous le fichier en version 2.

Maintenant, on s'imagine dans cette situation, placé sur le commit D. Notre fichier file.txt est en version 2.

Nous souhaitons supprimer le commit D. Trois stratégies existent.

La première déplace HEAD sur le commit parent de D, à savoir C. Le commit D n'est plus visible sur cette branche mais ni l'index ni le Working Directory ne sont modifiés.

Ce qui signifie un fichier toujours en version 2 dans l'éditeur et le fichier dans la zone verte du git status.

La seconde déplace toujours HEAD, mais modifie aussi l'index. Le Working Directory contient toujours file.txt en version 2.

Si on exécute git status, on trouvera alors file.txt dans la zone rouge.

La troisième déplace HEAD, modifie l'index mais aussi le Working directory !

  • # On garde les modifs, indexées
git reset HEAD~ --soft
  • # On garde toujours les modifs mais non indexées
git reset HEAD~ --mixed
git reset HEAD~
  • # On supprime toutes les modifs
git reset HEAD~ --hard

Pour résumer...

On note que l'option --mixed est équivalente à "aucune option" pour git reset.

Comment je reviens à la version du repository distant ?

Oups, j'ai tout cassé !

Bon pas de panique, j'avais un code fonctionnel sur le repository distant. Il faut juste que je me replace dessus...

Avant de parler de la solution, on va d'abord tenter de comprendre comment Git gère ses branches.

Le repository distant est symbolisé par le nuage. En local, on distingue deux types de branches :

- Les branches de travail, sur lesquelles on peut agir. Ce sont les branches que l'on manipule jusqu'à présent.

- Les branches de tracking, en lecture seule : ce sont des copies à un instant T du repository distant.

On agit sur les branches de tracking en communiquant avec le repository. Pull, fetch, push...

Pull ? Fetch ?

  • git pull origin master
  • git fetch origin master
git merge master

Mais au fait, fetch, pull, c'est quoi la différence ?

Un fetch met simplement à jour les branches de tracking pour les faire correspondre avec le repository distant. Les branches de travail ne sont pas modifiées.

Un pull n'est ni plus ni moins qu'un fetch suivi d'un merge. Lorsqu'on fait un pull -r (stratégie rebase), le merge est remplacé par un rebase.

  • git fetch origin feature1
git reset --hard origin/feature1

Pour revenir à la version de la branche distante, il suffit donc de se replacer au même niveau que la branche de tracking, après l'avoir mise à jour.

Le fetch met à jour la branche de tracking, le reset --hard replace le pointeur de notre branche actuelle sur celui de la branche de tracking.

En effet, une branche n'est ni plus ni moins qu'un pointeur vers un commit...

  • # A placer dans le .gitconfig
[alias]
  dammit = !BRANCH=$(git rev-parse --abbrev-ref HEAD) \
    && git fetch origin $BRANCH \
    && git reset --hard origin/$BRANCH
# Utilisation : git dammit

Pour faciliter la procédure, on peut écrire un alias Git.

Ah oui en effet... Au moment où j'ai commencé à développer ma feature, j'avais besoin de ce que Fabrice avait fait. Le problème c'est que son code n'était pas encore sur master. J'ai donc dû me baser sur sa branche.

Depuis, le code de Fabrice a été mergé, du coup je dois réappliquer mes commits en me basant sur master, pas sur l'ancienne branche de Fabrice !

On voit ici que Fabrice était parti de master pour développer sa feature.

Moi, j'avais besoin son code. Du coup je suis parti de la branche de Fabrice.

Mais entre temps, Fabrice a mergé son code. Mais ma branche se base toujours sur la branche de Fabrice et non sur master !

Dans ce cas là, si je fais un simple rebase sur master, aucun souci, Git sait s'y retrouver.

Par contre il existe un cas où c'est problématique : Si Fabrice a réécrit ses commits entre temps. Je me retrouve avec des anciens commits de Fabrice qui n'ont plus lieu d'être et dont je dois absolument me débarraser !

Je pense que maintenant vous vous souvenez de l'histoire des commits en double dont on a parlé précédemment... C'est le même risque ici.

    • # On n'oublie pas le ~ tilde ~ !
git rebase E~ --onto master
    • E~ : On réécrit à partir de E
    • --onto master : Au dessus de master

Donc à la fin, on souhaite arriver à ce résultat

Comme pour tout à l'heure, réécrire des commits signifie se replacer dans le passé, juste avant le premier commit à réécrire, donc on utilise encore un ~ tilde ~, qui pointe sur le parent du commit désigné.

L'option --onto désigne la branche sur laquelle se baser pour réappliquer ces commits.

Dans ce cas précis, Git va appliquer les commits E et F sur la base de master, donc il y a réécriture car changement de parent. Leurs sha1s vont donc changer en E' et F'.

Comment voir les modifications en cours ?

Première chose, comment je fais pour savoir ce que j'ai modifié, ce qui est prêt à être commité et ce qui ne l'est pas ?

  • # Modifications non indexées
git diff
  • # Modifications indexées
git diff --cached
git diff --staged
  • # Toutes les modifications
git diff HEAD
  • # Les modifications entre deux commits / branches
git diff ab0f..89df
git diff ab0f..master
git diff ab0f..HEAD

Vous aurez sûrement deviné... git diff ! Sauf que cette commande seule ne répertorie les modifications que sur les fichiers non encore dans l'index Git. Si j'ai déjà fait un git add, je ne vois plus mes modifs.

Pour cela, il y a les options --cached ou --staged, qui sont équivalentes.

Si je veux voir toutes les modifs, peu importe sur j'ai fait un git add ou non, j'utilise git diff HEAD. C'est le delta entre mon working directory et le dernier commit.

On peut même aller plus loin et demander les modifications entre 2 sha1, ou entre un sha1 et notre état actuel. N'hésitez pas à jouer avec !

Astuce suivante...

Je veux séparer les modifications d'un même fichier dans deux commits différents.

Tiens, ça c'est intéressant, et plutôt facile à mettre en oeuvre.

  • # pour indexer des modifications
git add -p
  • # Dans l'autre sens
git reset -p

L'option -p permet d'entrer dans un mode interactif où Git va nous proposer d'indexer ou non les modifications par petits morceaux appelés hunks.

Ce mode est aussi disponible pour la commande reset.

Démo !

La suite !

Je veux revenir sur la branche sur laquelle j'étais juste avant.

Une commande tellement simple et tellement méconnue...

  • # Comme "cd -" en shell !
git checkout -

De la même manière que cd - nous renvoie vers le dossier précédent en shell (comme le bouton précédent sur Windows !), cette commande Git nous renvoie vers la dernière branche sur laquelle on se trouvait.

Encore !

Je veux annuler les modifications faites sur un fichier en particulier.

Ici même chose, cela paraît simple, et peut-être qu'on utilise la syntaxe sans savoir pourquoi.

  • # Cette méthode a des défauts...
git checkout monFichier.ext
  • # Tout ce qui suit le "--"
# est considéré comme un nom de fichier
git checkout -- monFichier.ext

Très bien, cette méthode fonctionne la plupart du temps...

Mais qu'est-ce qui se passe si mon fichier s'appelle master ?

Ici le -- n'est pas obligatoire dans tous les cas mais très conseillé.

Si un fichier porte le même nom qu'un nom de branche, il risque d'y avoir conflit.

En shell, un double tiret signifie toujours la fin des options de la commande. Ce qui suit ici représente un nom de fichier. Cela permet d'éviter les malentendus.
